# 웹앱 운영 및 수익화 전략 (Web App Strategy)

> **목표**: 모바일 앱(iOS) 출시와 발맞추어, 기존 웹앱(`mandaact.vercel.app`)의 역할을 재정의하고 수익화 충돌 및 유지보수 문제를 해결한다.

---

## 1. 현황 및 문제점 (Current Situation)

| 항목 | 모바일 앱 (iOS) | 웹앱 (Vercel) | 이슈 (Issue) |
| :--- | :--- | :--- | :--- |
| **상태** | 출시 임박 (v1.0) | 구버전 유지 중 (Legacy) | 기능 및 디자인 불일치 |
| **수익화** | 부분 유료 (광고 + 구독) | 전면 무료 | **수익 간섭 (Cannibalization)**: 웹에서 무료로 프리미엄 기능(무제한/광고없음) 사용 가능 |
| **법적요건** | 웹 URL로 연결 | 약관/개인정보 처리방침 호스팅 | 앱스토어 심사를 위해 해당 URL(`privacy`, `terms`) 유지 필수 |

## 2. 단기 전략: "랜딩 페이지 모드" 전환 (즉시 실행)

**목표**: 수익 누수를 막고, 사용자 혼란을 방지하며, 앱스토어 심사 요건을 충족한다.

### 💡 전문가 제안 (Recommendation):
웹앱의 '앱 기능' 접근을 차단하고, **제품 소개용 랜딩 페이지(Marketing Landing Page)**로 전환합니다. 단, 법적 문서는 그대로 유지합니다.

#### 실행 방안 (Action Items):
1.  **메인 화면 (`/`) 개편**:
    *   기존 대시보드/로그인 화면 → **제품 소개 페이지**로 교체
    *   앱 주요 기능 소개, 스크린샷, "App Store에서 다운로드" 배너 배치
    *   **지원(Support) 정보 명시**: `support@unwrittenbd.com` 등 고객 지원 이메일 하단 표시 (앱스토어 마케팅 URL 요건 충족)
2.  **앱 기능 경로 차단 (`/app`, `/login` 등)**:
    *   **방안 A (접근 차단)**: 해당 경로 접속 시 메인(`/`)으로 리다이렉트 처리. 헤더의 '로그인/회원가입' 버튼 제거.
    *   **방안 B (안내 모달)**: 접속은 되나, *"모바일 앱 출시 준비 중입니다. iOS 앱을 이용해주세요."* 라는 공지 표시.
    *   👉 **방안 A(완전 차단)**를 추천합니다. 사용자가 "무료 웹 버전"을 찾아내는 것을 사전에 방지합니다.
3.  **필수 경로 유지**:
    *   `/privacy` (개인정보처리방침) 및 `/terms` (이용약관) 페이지는 **반드시 접근 가능**해야 합니다. (기존 유지)

**기대 효과:**
*   "웹에서는 공짜던데?"라는 우회를 차단하여 모바일 구독 전환율 증대.
*   구버전 웹 사용으로 인한 고객 문의 및 데이터 오류 방지.
*   앱스토어 다운로드 유도 집중 및 SEO(도메인 권위) 유지.

## 3. 장기 전략: "프리미엄 컴패니언" 모델 (유료화 연계)

**목표**: 웹앱을 '무료 대체재'가 아닌 **'유료 사용자를 위한 확장 기능'**으로 포지셔닝하여 구독 가치를 높인다.

### 💡 전문가 제안 (Recommendation):
웹앱을 **프리미엄(구독) 사용자 전용 기능**으로 운영합니다. (e.g., "PC에서도 편하게 만다라트 편집 가능")

#### 운영 모델 (Companion Model):
1.  **진입점**: 모바일이 메인. 결제는 모바일 IAP(인앱결제) 위주로 유도.
2.  **웹 접근 권한**:
    *   iOS에서 '프리미엄'을 구독한 계정으로 웹 로그인 시 → **전체 기능 사용 가능**.
    *   무료 사용자 로그인 시 → **보기 전용(Read-only)** 또는 **"편집하려면 프리미엄이 필요합니다"** 안내.
3.  **크로스 플랫폼 가치**:
    *   생산성 앱 특성상, 키보드/큰 화면 활용 니즈가 큼.
    *   "언제 어디서나(Anywhere) 접속"을 프리미엄 혜택으로 마케팅.

#### 기술적 준비사항:
*   모바일/웹 공통 DB(Supabase)를 사용하므로, `user_subscriptions` 테이블을 조회하여 웹 접근 제어 가능.
*   웹앱 UI를 모바일 최신 디자인(Sub-goals 등)에 맞춰 업데이트 필요 (데이터 구조 통일).

#### ⚠️ 구독 계정 보안 전략 (Risk Management):
*   **문제점**: 앱스토어 ID(결제)는 하나인데 앱 계정(이메일)을 여러 개 생성하여 '구매 복원'을 반복하면, 1개의 결제로 다수의 웹 계정이 생성될 우려가 있습니다.
*   **해결책**: RevenueCat의 **'Transfer to new ID' (소유권 이전)** 정책을 적용합니다.
    *   원칙: **"동시에 오직 하나의 앱 계정(Supabase ID)만 프리미엄 권한을 보유한다."**
    *   시나리오: 사용자가 A계정에서 결제 후 B계정으로 '구매 복원'을 시도하면, **A계정의 프리미엄 권한은 즉시 회수(만료)**되고 B계정으로 이전됩니다.
    *   **장점 (Implementation Benefit)**: 이 정책 변경은 **서버(RevenueCat 대시보드) 설정**만으로 적용 가능하므로, **모바일 앱의 재빌드/재심사/업데이트가 불필요**합니다.
    *   **결과**: 장기적으로도 유효하며, 웹앱 무단 공유(어뷰징)를 원천적으로 차단할 수 있습니다.
